# Agent Hooks

Automate agent behavior with event-driven hooks that trigger actions at key points during a session.

## What It Does

Agent Hooks let you attach automatic actions to events in the agent's lifecycle. When the agent writes a file, finishes a turn, or tries to stop, your hooks can run linters, capture screenshots, block dangerous operations, or advance multi-phase workflows -- all without manual intervention.

Hooks run silently in the background. When a hook produces output (like lint errors), that feedback is injected directly into the agent's context so it can act on it immediately.

## Setting Up Hooks

1. Open the **Agent** view
2. Click the **Session Config** menu (gear icon in the toolbar)
3. Select **Hooks** to open the Hooks panel
4. You can either:
   - Click **Templates** to pick a pre-built hook
   - Click the **+** button to create a custom hook from scratch
5. After adding hooks, toggle the global **Hooks ON / Hooks OFF** button to enable or disable all hooks at once

Each hook has its own individual toggle as well, so you can enable or disable specific hooks without removing them.

Hooks are persisted across sessions. They apply to whichever agent session is active.

## Triggers

Every hook is attached to a **trigger** -- the event that causes it to fire.

### Tool Completed

Fires after a tool finishes executing. Use this for post-action checks like linting a file that was just written, or capturing a screenshot after a UI change.

When combined with a **tool filter**, this trigger only fires for specific tools. For example, you can lint only after `write_file` and `edit_file`, ignoring `read_file` or `search_code`.

### Iteration End

Fires after each full agent turn (one round of thinking + tool calls). This is the right place for broader checks that apply to the whole iteration, like running `tsc --noEmit` to type-check the entire project, or running `cargo check`.

### Session End

Fires when the agent session finishes. Use this for cleanup, final reports, or notifications (e.g., "Agent session complete").

### Pre-Tool Use

Fires **before** a tool runs. This trigger is special because it can **block** the tool from executing. Use it as a guardrail -- for example, preventing the agent from modifying `.env` files or writing to protected directories.

Pre-Tool Use hooks support tool filters so they only intercept specific tools.

### Stop

Fires when the agent decides to stop. This trigger can **override the stop** and force the agent to continue with a new prompt. This is the foundation for multi-phase workflows: when the agent finishes one phase, the stop hook checks if there are more phases and injects the next phase's prompt.

## Actions

Each hook performs one **action** when it fires.

### Screenshot

Captures a screenshot and adds it to the agent timeline. Two modes:

- **Desktop capture** (default) -- takes a screenshot of your screen using the system screenshot tool
- **URL capture** -- renders a web page in headless Chrome and screenshots it (useful for previewing localhost during development)

Options:
- **URL** -- leave blank for desktop capture, or enter a URL (e.g., `http://localhost:5173`)
- **Delay (ms)** -- wait before capturing (useful to let animations or page loads finish)

### Command

Runs a shell command in your sandbox directory. The command's output is captured, and depending on the exit code, it can inject feedback into the agent or block a tool call.

Options:
- **Command** -- any shell command (e.g., `cargo check 2>&1 | tail -20`)
- **Timeout** -- maximum seconds to wait (default 30, max 120)

See the [Advanced: Custom Command Hooks](#advanced-custom-command-hooks) section for the exit code protocol.

### Notify

Shows a notification. Useful for alerting you when something happens -- the agent finished, a specific tool was called, etc.

Options:
- **Message** -- the notification text

### Lint

Runs a linter on your project and feeds any errors back to the agent. The agent sees the errors as context and can fix them immediately.

Options:
- **Linter** -- choose a specific linter or leave on **Auto-detect**:
  - **Auto-detect** checks your project for config files in this order: Biome (`biome.json`) > ESLint (`.eslintrc.*` or `eslint.config.*`) > Cargo (`Cargo.toml`) > TypeScript (`tsconfig.json`)
  - **Biome** -- fastest option (sub-second per file)
  - **ESLint** -- the standard JavaScript/TypeScript linter
  - **TypeScript (tsc)** -- runs `tsc --noEmit` for type checking (best for Iteration End triggers, not per-file)
  - **Cargo** -- runs `cargo check` for Rust projects
- **Auto-fix** -- when checked, attempts to auto-fix issues (where the linter supports it)

Lint results are truncated for token efficiency. Critical errors (type errors, undefined references, security issues) are always included. Minor style issues are filtered when there are many errors.

### Workflow

Reads the workflow state file and advances through phases. When the agent tries to stop, this action checks if there are remaining workflow phases. If so, it forces the agent to continue with the next phase's prompt.

Workflows are created in the **Workflow** panel (Session Config > Workflow). Each workflow defines a series of phases with names and prompts. The workflow action manages phase transitions, tracks completion status, and respects budget limits.

## Templates

The Hooks panel includes pre-built templates you can add with one click.

| Template | Trigger | Action | What It Does |
|----------|---------|--------|--------------|
| **Lint on write** | Tool Completed | Lint | Runs Biome or ESLint whenever a file is written or edited. Feeds errors back to the agent. |
| **Type check per iteration** | Iteration End | Lint (tsc) | Runs `tsc --noEmit` after each agent turn. Catches type errors across the whole project. |
| **Screenshot on write** | Tool Completed | Screenshot | Captures a screenshot whenever a file is written. Useful for visual verification of UI changes. |
| **Screenshot on iteration** | Iteration End | Screenshot | Captures a screenshot after each agent turn. |
| **Block .env writes** | Pre-Tool Use | Command | Prevents the agent from modifying any `.env` file. The tool call is blocked with a reason. |
| **Multi-phase workflow** | Stop | Workflow | Keeps the agent running through all workflow phases instead of stopping after the first one. |
| **Build check on iteration** | Iteration End | Command | Runs `cargo check` after each iteration. Shows the last 20 lines of output. |

## Advanced: Custom Command Hooks

Command hooks use an **exit code protocol** to communicate decisions back to the agent. The behavior depends on which trigger the hook is attached to.

### Pre-Tool Use Commands

When a command hook is attached to the Pre-Tool Use trigger, the exit code determines what happens to the tool call:

| Exit Code | Behavior |
|-----------|----------|
| **0** | Approve -- the tool executes normally |
| **1** | Block -- if stderr starts with `BLOCK:`, the tool is blocked and the reason is shown to the agent |
| **2** | Inject context -- stderr content is added to the agent's context (the tool still executes) |

The hook receives two environment variables:
- `HOOK_TOOL_NAME` -- the name of the tool being called (e.g., `write_file`)
- `HOOK_TOOL_ARGS` -- the tool's arguments as a string

Example: blocking writes to `.env` files:

```sh
case "$HOOK_TOOL_ARGS" in
  *".env"*) echo "BLOCK: Cannot modify .env files" >&2; exit 1;;
esac
exit 0
```

### Stop Commands

When a command hook is attached to the Stop trigger, the exit code determines whether the agent continues:

| Exit Code | Behavior |
|-----------|----------|
| **0** | Allow -- the agent stops normally |
| **2** | Force continuation -- stderr (or stdout if stderr is empty) becomes the new prompt injected into the agent |

Example: forcing the agent to run tests before stopping:

```sh
RESULT=$(cd "$SANDBOX_PATH" && npm test 2>&1)
if echo "$RESULT" | grep -q "FAIL"; then
  echo "Tests are failing. Fix these failures before stopping: $RESULT" >&2
  exit 2
fi
exit 0
```

### Tool Completed / Iteration End Commands

For these triggers, exit code 2 with stderr content injects that content as feedback to the agent. Exit code 0 means no feedback is injected.

## Tips

- **Start with templates** -- the pre-built hooks cover the most common use cases. Customize from there.
- **Use tool filters** -- a "lint on write" hook should only trigger on `write_file` and `edit_file`, not on every tool call. Tool filters are available for Tool Completed and Pre-Tool Use triggers.
- **Lint action vs. command action** -- the Lint action auto-detects your linter, formats output for token efficiency, and handles errors gracefully. Prefer it over manually running lint commands.
- **Be careful with Stop hooks** -- a stop hook that always forces continuation can create an infinite loop. Make sure your stop condition is reachable (e.g., all workflow phases completed, or tests passing).
- **Timeout awareness** -- command hooks have a default 30-second timeout (max 120 seconds). Long-running commands like full project builds should set an appropriate timeout.
- **Hook output is truncated** -- to save tokens, command output is limited to about 2000 characters and lint reports to about 1500 characters. Critical errors are prioritized.
- **Global toggle** -- use the Hooks ON/OFF button to quickly disable all hooks without removing them. Individual hooks also have their own toggles.
- **Hooks persist** -- your hook configuration is saved and restored across app restarts.

## Related Pages

- [Agent](agent.md) -- the core agentic AI system that hooks extend
- [Terminal](terminal.md) -- interactive terminal paired with agent sessions
- [Roles & Skills](roles-and-skills.md) -- custom agent profiles and system prompts

# Agent

The agentic AI assistant that can read and write files, execute commands, search code, generate media, and operate autonomously within your project.

## What It Does

The Agent is the core feature of Vibing with Grok. Unlike the Chat page (which is a standard conversation), the Agent has full access to tools that let it interact with your filesystem, run shell commands, analyze codebases, generate images and videos, and more. You give it a task, and it figures out what tools to use to accomplish it -- reading files to understand context, writing code, running tests, and iterating until the job is done.

Every agent session is scoped to a **sandbox path** -- a project directory on your machine. The agent can only read and write files within this directory (and its subdirectories). This keeps it focused on your project and prevents it from wandering into unrelated parts of your system. As the agent works, each tool call appears in a real-time timeline so you can see exactly what it is doing at every step.

The agent uses a permission system to keep you in control. Safe, read-only operations (like reading a file or listing a directory) execute automatically. Operations that modify your project (writing files, running commands) require your explicit approval through a permission modal -- unless you opt into Skip All mode for hands-free operation. Dangerous operations (like `rm -rf` or modifying system files) are always blocked regardless of mode.

## Getting Started

1. **Navigate to Agent** -- click "Agent" in the sidebar, or press the keyboard shortcut if configured.

2. **Set your sandbox path** -- in the toolbar at the top, enter the path to the project directory you want the agent to work in. You can:
   - Type or paste a path directly into the text field
   - Click the folder icon to open a file browser
   - Drag and drop a folder from Finder onto the path field

3. **Choose a model** -- use the model dropdown in the toolbar. Available models include:
   - **Grok 4.1 Reasoning** (`grok-4-1-fast-reasoning`) -- the default, with chain-of-thought reasoning
   - **Grok 4.1 Fast** (`grok-4-1-fast-non-reasoning`) -- faster responses without reasoning traces
   - **Grok Code** (`grok-code-fast-1`) -- optimized for coding tasks

4. **Type your request** -- describe what you want done in the message input at the bottom. Press Enter or click the send button.

5. **Monitor the timeline** -- the agent streams its response in real time. Each tool call appears as a card in the timeline showing the tool name, arguments, and result.

6. **Handle permissions** -- when the agent wants to write a file or run a command, a permission modal appears. Review the action and click Approve or Deny. The modal shows exactly what the agent wants to do, including the full command or file path.

## Key Features

### Permission System

Tools are classified into three tiers:

- **Whitelisted (auto-approved)** -- read-only and safe operations execute without interruption. This includes reading files, listing directories, searching code, generating images/videos, memory operations, code graph queries, and collection searches.
- **Graylisted (requires approval)** -- operations that modify your project show a permission modal before executing. This includes writing files, editing files, running shell commands, executing code, deleting files, uploading to collections, and ingesting code graphs.
- **Blacklisted (always blocked)** -- dangerous patterns are hard-blocked regardless of any setting. This includes destructive shell commands (`rm -rf /`, `mkfs`, `dd`), interactive/blocking commands (`vim`, `nano`, `top`), and writes to sensitive paths (`.env`, credentials, SSH keys).

Use the **Skip All** toggle (shield icon in the toolbar) to auto-approve all graylist tools. This is useful when you trust the agent and want maximum speed, but use it with care.

### Git Checkpoints

When the agent modifies files, the app automatically creates git commits after each write operation. This gives you a checkpoint timeline showing every change the agent made during the session.

- **Session branch** -- each agent session starts on a dedicated git branch, keeping agent changes isolated from your main work.
- **Checkpoint timeline** -- click the git branch icon to see all checkpoints with timestamps and commit messages.
- **Revert** -- click the revert button on any checkpoint to reset your files to that point. You can also revert all the way back to the session start.
- **Confirmation** -- reverts always ask for confirmation before resetting files.

### Session Tabs

Run multiple agent sessions in parallel using the tab bar above the agent view.

- Click the **+** button to create a new session (uses the current sandbox path).
- Click a tab to switch between sessions.
- **Double-click** a tab to rename it. The paired terminal (if any) is also renamed automatically.
- Click the **X** on a tab to close it.
- A small terminal icon appears on tabs that have an active terminal session.
- Use the **history** button to browse and restore previous sessions.

### View Modes

The toolbar provides three view modes:

- **Agent** -- the default view showing the conversation timeline and tool calls.
- **Split** -- side-by-side view with the agent timeline on one side and an integrated terminal/browser on the other.
- **Terminal** -- full-screen terminal view paired with the current agent session.

When the agent runs a shell command, a terminal session is automatically created. A pulsing dot appears on the Terminal button to indicate activity.

### Chat Assistant

A secondary chat panel that opens alongside the agent. Use it to ask quick questions, brainstorm, or get explanations without interrupting the agent's workflow. Toggle it with the "Chat Assistant" button in the toolbar.

### Sub-Agents

The agent can spawn lightweight sub-agents for parallel research tasks. Sub-agents:

- Use a faster model for quick lookups
- Have read-only tool access (they cannot modify files or run commands)
- Appear as collapsible cards in the timeline showing their own tool calls and findings
- Are limited to 6 per session with 8 iterations each

### Plan Mode

Enable Plan mode from the session options menu (+ button near the message input). When active, the agent first creates a structured plan before executing any actions. This is useful for complex multi-step tasks where you want to review the approach before the agent starts making changes.

### Incognito Mode

Toggle Incognito from the session options menu. When active, the session is not saved to history and no session logs are written. Use this for sensitive work you do not want persisted.

### Agent Memory

The agent can save and recall facts across sessions. Memory facts are persistent -- they survive app restarts and are available in future sessions. The agent uses two memory tools:

- **save_memory_fact** -- stores a fact (e.g., "This project uses pnpm, not npm")
- **search_memories** -- retrieves relevant facts based on a query

You can toggle memory on or off from the session options menu. When memory is disabled, the agent will not read or write facts for that session.

### Context Files

Attach specific files for the agent to reference at the start of a conversation. Click "Attach files" from the session options menu or drag files onto the input area. Attached files appear as pills above the input and their contents are included in the agent's context.

### MCP Integration

Connect external tool servers using the Model Context Protocol. Open the MCP Servers panel from the session options menu to add server configurations. Connected MCP servers extend the agent with additional tools (e.g., browser automation, knowledge bases, external APIs). The count badge shows how many servers are connected.

### Hooks

Automate agent behavior with hooks -- rules that trigger on specific events. Open the Hooks panel from the session options menu. Example hooks include:

- Lint-on-save: automatically run a linter after the agent writes a file
- Tool blocking: prevent the agent from using certain tools
- Multi-phase workflows: chain multiple agent actions together

See [Agent Hooks](agent-hooks.md) for full documentation.

### Toolkit

Manage which tools are available to the agent. Open the Toolkit panel from the session options menu to enable or disable specific tools. This is useful when you want to restrict the agent to a subset of its capabilities.

### Batch Processing

Submit large-scale processing jobs directly from the agent. Open the Batches panel from the session options menu. The agent can also create and manage batch jobs programmatically using its batch tools.

### Cost Tracking

A live cost badge appears in the toolbar showing the running cost of the current session. Click it to see a breakdown of input and output tokens with per-category costs.

# Arena

Git worktree management for isolated feature development and parallel experimentation.

## What It Does

The Arena page gives you a visual interface for creating and managing git worktrees -- lightweight, isolated copies of your repository that live on separate branches. Each worktree is a fully functional checkout of your project, but changes in one worktree do not affect any other worktree or your main branch. This makes it safe to experiment, prototype, or run multiple agent tasks in parallel without worrying about conflicts or breaking your working copy.

Under the hood, git worktrees share the same repository history and objects, so creating one is fast and uses minimal disk space compared to cloning the entire repository. When you create a worktree in the Arena, you get a new directory with its own branch where you (or the agent) can make changes freely. When you are done, you can merge the branch back into your main branch, or simply delete the worktree to discard the experiment.

The Arena is especially powerful when combined with the agent. You can spin up multiple worktrees and run separate agent sessions in each one, all working on different tasks simultaneously. For example, you might have one worktree for a bug fix, another for a new feature, and a third for a refactoring experiment -- all running in parallel, all completely isolated from each other. The task queue's worktree executor takes this further by automatically assigning queued tasks to their own worktrees for hands-free parallel execution.

## Getting Started

1. Navigate to the **Arena** page (`/arena`) from the sidebar.
2. Make sure you have a git repository set as your sandbox path. The Arena requires an initialized git repository to create worktrees.
3. Click **Create Worktree** to create a new isolated workspace.
4. Give the worktree a descriptive name (for example, "fix-login-bug" or "experiment-new-ui"). This name is used for both the worktree directory and the new branch.
5. The Arena creates a new branch based on your current HEAD and checks it out in a separate directory.

# MCP Servers

Connect external tool servers via the Model Context Protocol to extend what the agent can do.

## What It Does

MCP (Model Context Protocol) is a standard that lets AI models interact with external services through tool interfaces. When you connect an MCP server to Vibing with Grok, the agent gains access to whatever tools that server exposes -- searching databases, reading from cloud storage, interacting with third-party APIs, controlling a browser, or anything else the server provides.

You configure MCP servers on this page, and they become available in all agent sessions (as long as MCP is enabled). Each server is identified by its URL, and you can optionally provide an authentication token, a human-readable label, a description, and a list of allowed tool names to restrict which tools the agent can use from that server. The app communicates with MCP servers over HTTP, so any server reachable by URL will work -- whether it is running locally on your machine, on your network, or hosted remotely.

You can also configure MCP servers on a per-session basis from the agent toolbar. The per-session configuration is useful when you want a server available only for a specific task without adding it to your global list. A global enable/disable toggle lets you turn off all MCP servers at once without removing them from the list.

## Getting Started

1. **Navigate to MCP Servers** -- click "MCP" in the sidebar under Dev Tools, or go to `/mcp`.

2. **Click "Add Server"** -- a form appears where you enter the server details.

3. **Enter the Server URL** -- this is the HTTP endpoint of the MCP server (for example, `https://your-mcp-server.com/mcp`).

4. **Optionally fill in other fields**:
   - **Label** -- a short name for the server (e.g., "my-tools"). If left empty, the hostname is used.
   - **Auth Token** -- a bearer token if the server requires authentication.
   - **Description** -- a note about what this server provides.
   - **Allowed Tool Names** -- a comma-separated list of specific tools to allow. Leave empty to allow all tools the server offers.

5. **Click "Add Server"** -- the server appears in your installed servers list.

6. **Test the connection** -- click the lightning bolt icon on the server card to run a health check. If the server is reachable, it will show a green status dot, the latency, and a list of discovered tools.

7. **Start an agent session** -- the agent now has access to the tools from your configured MCP servers. It will use them when relevant to your request.

## Key Features

- **Global server registry** -- add servers once and they are available across all agent sessions.
- **Health checks** -- test server connectivity with a single click. See reachable/unreachable status, latency, and error messages. Use "Check All" to test every server at once.
- **Tool discovery** -- when a health check succeeds, the app lists all tools the server exposes. Each tool appears as a labeled badge on the server card.
- **Allowed tool filtering** -- restrict which tools the agent can access from a server. This is useful when a server exposes many tools but you only want the agent to use a few.
- **Authentication support** -- provide a bearer token for servers that require it. The token is stored locally and sent with every request to that server.
- **Global enable/disable** -- turn all MCP servers on or off with a single toggle in the header. When disabled, agent sessions will not use any configured MCP servers.
- **Per-session configuration** -- open the MCP Servers panel from the agent toolbar to add servers that apply only to the current session.
- **High-risk detection** -- the app automatically flags servers whose URL, label, or description contains keywords related to email or messaging services (e.g., "gmail", "slack", "smtp", "twilio"). These servers require you to explicitly acknowledge the security risk before they can be added.
- **Expandable server cards** -- click the chevron on any server card to see its full URL, auth status, and allowed tools. Health check results appear here too.

## Security

MCP servers give the AI access to external services. This is powerful but comes with risk. The app includes built-in safety guidance:

- **Prompt injection warning** -- servers that handle email, messaging, or similar services are flagged because an attacker could send you a message containing hidden instructions that the AI might follow. This is not theoretical -- it is one of the most practical attack vectors against AI systems with tool access.
- **Automatic risk detection** -- servers are labeled "HIGH RISK" if their URL, label, or description matches email or messaging keywords. Adding a high-risk server requires an explicit acknowledgment.
- **Safe usage guidelines** -- only connect servers you trust, use allowed tool name filters to minimize the attack surface, prefer read-only servers when possible, and review discovered tools before use.

## Tips

- **Start with read-only servers.** If you are trying MCP for the first time, connect a server that only provides search or read operations. This minimizes risk while you get comfortable with the workflow.

# Sandbox

Run agent tools inside isolated containers so the agent can work freely without affecting your host machine.

## What It Does

The Sandbox wraps the agent's filesystem and shell tools in a container (Docker or Podman). When the sandbox is active, every file write, shell command, and code execution the agent performs happens inside the container rather than directly on your machine. Your project directory is mounted into the container at `/workspace`, so the agent can still read and modify your project files, but everything else -- system files, other directories, installed packages -- is isolated.

The key benefit is permission escalation: when the sandbox is running, tools that normally require your approval (writing files, running commands, executing code) are auto-approved because the blast radius is contained. This lets you give the agent free rein to iterate rapidly on a task without clicking through permission modals, while knowing that any mistakes are confined to the container. API-based tools (like collections, trading, or batch processing) pass through unchanged since they do not touch the local filesystem.

The sandbox also enforces security hardening. Containers run with dropped capabilities and no-new-privileges restrictions. Even inside the sandbox, the app's blacklist rules still apply -- dangerous commands and sensitive file paths are blocked for defense in depth.

## Getting Started

1. **Install Docker or Podman** -- you need a container runtime on your machine. The Sandbox page auto-detects which runtimes are available.

2. **Navigate to Sandbox** -- click "Sandbox" in the sidebar under Dev Tools, or go to `/sandbox`.

3. **Check available runtimes** -- the Status tab shows detected runtimes (Docker, Podman) and whether their daemons are running. If a daemon is not running, click "Start" next to it.

4. **Configure the sandbox** -- switch to the Configure tab:
   - **Backend type** -- choose Container (Docker/Podman) or Firecracker (coming soon, Linux only).
   - **Host directory** -- set the project directory to mount into the container. Click "Browse" to pick a folder.
   - **Container runtime** -- select Docker or Podman.
   - **Container and image names** -- defaults are usually fine (`vibing_sandbox` and `vibing-sandbox:latest`).
   - **CPU and memory limits** -- adjust how many cores and how much RAM the container can use.
   - **Network mode** -- choose "Bridge" for internet access or "None" for full network isolation.
   - Click **Save Configuration**.

5. **Build the sandbox image** -- switch to the Image tab and click "Build Image". This creates a container image with common development tools (gcc, Python, Node.js, Rust, ripgrep, and more). Build output streams in real time. You only need to do this once.

6. **Start the sandbox** -- go back to the Status tab and click "Start Sandbox". A green "Running" badge appears in the header, and the uptime counter begins.

7. **Use the agent normally** -- start an agent session. All filesystem and shell operations now execute inside the container. The sandbox status indicator appears in the agent toolbar so you know it is active.

## Key Features

- **Container isolation** -- agent file writes and shell commands run inside a Docker or Podman container, not on your host.
- **Auto-approved writes** -- graylist tools (write_file, edit_file, execute_command, execute_code) skip the permission modal when the sandbox is active because changes are contained.
- **Bind-mounted workspace** -- your project directory is mounted at `/workspace` inside the container, giving the agent full read/write access to your project while isolating everything else.
- **Runtime auto-detection** -- the Status tab automatically detects Docker and Podman installations and shows whether their daemons are running.
- **Daemon management** -- start a container runtime daemon directly from the Status tab if it is installed but not running.
- **Resource limits** -- configure CPU core count (1-16) and memory limit (2GB-32GB) to control how much of your machine the container can use.
- **Network modes** -- "Bridge" gives the container internet access (needed for package installs, API calls). "None" fully isolates the container from the network.
- **Security hardening** -- containers run with `--cap-drop=ALL` plus a minimal set of required capabilities, and `--no-new-privileges` prevents privilege escalation.
- **Defense in depth** -- blacklist checks (blocked paths, dangerous commands) still apply inside the sandbox.
- **Pre-built image** -- the Image tab builds a sandbox image with gcc, Python 3, Node.js, Rust, ripgrep, fd, tree, jq, and other essential development tools.
- **Status monitoring** -- see container name, backend type, and uptime at a glance. The status auto-refreshes every 10 seconds while running.
- **Firecracker preview** -- a Firecracker MicroVM backend is planned for Linux systems with KVM support, providing hardware-level isolation via lightweight microVMs.

## Tips

- **Use the sandbox for exploratory tasks.** When asking the agent to "try different approaches" or "experiment with this", the sandbox lets it iterate without worrying about cleaning up failed attempts on your real filesystem.

- **Build the image once, reuse it.** The sandbox image only needs to be built once. It persists across app restarts. Rebuild only if you need to update the base tools.

- **Set resource limits thoughtfully.** If the agent will compile large projects or run test suites, give the container enough CPU and memory. For simple file editing tasks, minimal resources are fine.

- **Use "None" network mode for maximum isolation.** If you do not need the agent to download packages or make API calls from inside the container, disable networking entirely.

- **The sandbox indicator in the agent toolbar confirms activation.** If you do not see it, the sandbox may not be running -- check the Sandbox page.

- **Mission Control recommends the sandbox.** Mission Control requires sandbox to be active because its workers need auto-approved write permissions to execute tasks in parallel without blocking on permission modals.

- **API tools are not affected.** Collections, batch processing, trading, and other API-based tools work the same whether the sandbox is active or not -- they do not touch the local filesystem.

Because the planner cannot modify files, it stays focused on analysis rather than jumping into premature edits.

Each side of the Workspace is a full agent session with its own sandbox path, conversation timeline, and tool history. - **Planning agent (left panel)** -- read-only tool access. Can read files, list directories, search code, query the code graph, and search collections. Cannot write files, run commands, or make any modifications. This constraint keeps it focused on analysis and planning.

- **Execution agent (right panel)** -- full tool access. Can do everything the regular Agent can: write files, edit files, run shell commands, execute code, and use all other tools with the standard permission system.

- **Shared task board** -- the central coordination mechanism. The Planning agent creates tasks that describe what needs to be done. The Execution agent picks them up and marks them as completed. Both agents can see and reference the full task list.

- **Independent sessions** -- each panel maintains its own conversation history, tool timeline, and context. You can scroll through either panel's history independently.

- **Independent sandbox paths** -- while you will usually point both panels at the same project, having separate paths gives you flexibility for cross-project work or reference reading.

- **Real-time task updates** -- when the Planning agent adds a task or the Execution agent completes one, the task board updates immediately for both sides.

- **Full agent capabilities per panel** -- each side supports model selection, permission handling, git checkpoints (execution side), and all the standard agent features described in the Agent guide.

## Tips

- **Let the planner go first.** Give the Planning agent time to analyze before asking the executor to start implementing. A good plan saves time on the execution side.

- **Use the planner for code review.** After the Execution agent finishes a batch of changes, ask the Planning agent to review what was modified and flag any issues.

- **Break large tasks into phases.** Ask the planner to create tasks for phase one, let the executor finish them, then ask the planner to plan phase two based on the current state of the code.

- **Use different models per panel.** The Planning agent benefits from a reasoning model that thinks carefully about structure. The Execution agent can use a faster model for quicker turnaround on implementation tasks.

- **The planner cannot accidentally break anything.** Because it has read-only access, you can let it explore freely without worrying about unintended modifications.

- **Reference the task board in your prompts.** You can tell the executor something like *"Pick up the next pending task from the board"* or tell the planner *"Create tasks for the remaining work based on what the executor has completed so far."*
